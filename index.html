I. Introduction

Project brief.

The Apex Quantitative Investing Project is an internal research initiative combining quantitative analysis, multi-model AI consensus, and disciplined capital allocation. It is a startup-scale programme run by a team of 2–4 people. The pre-launch build window is 6–12 months; during this period the framework is developed, validated, and stress-tested before any operational deployment.

Governed by consensus.
Executed with discipline.

The project operates across two distinct portfolios — one mandated for long-horizon compounding, the other for short-horizon tactical capture. Every decision passes through a multi-stage pipeline in which data is cross-validated, knowledge is structured, quantitative models are run in parallel, and outputs are reconciled before any recommendation reaches the owner.

The mandate permits long-buying, short-buying (short positions), and leverage (e.g. leveraged or inverse ETFs), all executed as normal cash-settled transactions. Margin financing on the broker side and direct derivative contracts (options, futures, swaps) are excluded.

Portfolio construction rule.

Each portfolio holds ≤ 8 distinct instruments. Overlap is permitted between the long-term and short-term portfolios where the same instrument suits both mandates. Weights are not fixed — they are solved at each rebalance for the highest expected return at the minimum achievable risk, within the aggressive-investor risk budget.

Instrument selection rule.

Specific instruments are not pre-selected in this document. Candidate universes will be generated by the quantitative engine and presented to the owner at each rebalance. Final selections are made at that point — never in advance.

III. System Workflow

Six steps. Four automated.

Steps 1 through 4 run autonomously, twenty-four hours a day. Steps 5 and 6 require owner interaction. Every output — successful or otherwise — is written back to the knowledge base as training data, ensuring the system improves with each cycle.

Phase A · Automated (Steps 1 – 4)

24/7 autonomous operation. Data is ingested, cross-validated, stored, and analysed by the quantitative engine without human intervention.

Phase B · Owner-in-the-loop (Steps 5 – 6)

Owner consultation required. Quantitative analysis outputs are reviewed, discussed, approved, and fed back into the system with post-trade context.

01 / 06
Ingest
Auto · 24/7

Auto-import raw data from 10–15 external APIs.

Continuous ingestion of financial data, economic indicators, market trends, geopolitical events, and business news into a central raw data store. 13 sources total — 8 primarily for the long-term portfolio, 7 primarily for the short-term portfolio, with 2 explicitly shared across both books. See the “Data Sources” section above for the full source list with subscription links.

02 / 06
Validate
Auto · 24/7

Tri-model cross-validation and fake-news filter.

Raw data is cross-validated in parallel by the latest Claude (Anthropic), ChatGPT GPT-5.5 (OpenAI), and Gemini (Google) models. Each model independently scores credibility, flags contradictions, and eliminates unreliable sources.

Consensus decision logic
  • 3 of 3 agree → VALID. Data is committed to the knowledge base.
  • 2 of 3 agree → PENDING. Data enters a 30-minute re-check loop, repeated up to three times.
  • Loop exhausted & still pending → ELIMINATE.
  • Fewer than 2 of 3 agree → ELIMINATE immediately.
T+0Initial Check
T+30mRecheck 1
T+60mRecheck 2
T+90mFinal Verdict
Recommended models (April 2026) & anti-hallucination protocol
  • AnthropicClaude Opus 4.7 — highest-accuracy reasoning in the Claude 4.7 family; Sonnet 4.6 / Haiku 4.5 used for high-volume routine validation.
  • OpenAIChatGPT GPT-5.5 — natively omnimodal, 1M context window, current Artificial Analysis Intelligence Index leader (launched 23 April 2026); GPT-5.4 / GPT-5-mini routed for cheaper tasks.
  • GoogleGemini 3.1 Pro — strong reasoning benchmarks, multimodal; Flash 2.5 for fast routine checks.

How to command each model for accurate validation (before cross-checking):

  • 1Structured-output prompting. Force JSON schema with claim, source, credibility_score (0–1), red_flags array, reasoning. No free-text answers.
  • 2Chain-of-Verification (CoVe). The model generates verification questions about its own answer, answers them, and revises. Materially reduces hallucination.
  • 3Grounding only. Use only the provided source text — no external memory. Any unsupported claim is flagged.
  • 4Self-consistency. Call each model three times at temperature 0.3; take the majority vote within that model before cross-comparing.
  • 5Adversarial red-team prompt. Ask each model to “find reasons this source might be false or misleading.” Only data that survives proceeds.
  • 6Retrieval-Augmented Generation. Cross-reference claims against a vector index of previously validated facts.
  • 7Citation enforcement. Every factual assertion must cite a source URL or document identifier; uncited claims are rejected.
  • 8Temperature discipline. Validation calls run at 0.1–0.2; red-team calls run higher for divergence-seeking.
  • CostUse Batch API for non-urgent validation. All three providers offer Batch tiers at 50% of standard price — material savings on overnight filings or end-of-day news batches.
03 / 06
Memorise
Auto · 24/7

The second brain · knowledge database.

Validated data is categorised, tagged, and stored in a hybrid graph-and-vector knowledge base. Claude API discovers and maintains relationships between events, sectors, assets, and macro indicators. Two parallel brains are maintained — one per portfolio — with shared cross-portfolio data available to both.

Brain 1 · Long-Term

Fundamentals, macro regimes, sector rotations, secular trends, valuation histories. Also receives all dual-portfolio validated data.

Brain 2 · Short-Term

Intraday flows, momentum, volatility regimes, news sentiment, options flow, geopolitical shocks. Also receives all dual-portfolio validated data.

Startup-sized tooling for Step 3
  • CoreSupabase (Postgres + pgvector) — replaces both Neo4j Enterprise and Pinecone at this stage. Paid tier starts at USD 25/month. Scales to tens of millions of rows.
  • GraphApache AGE extension for relationship queries. Upgrade path to Neo4j AuraDB if graph-traversal performance becomes a bottleneck.
  • OrchestrationLlamaIndex + LangChain for RAG orchestration. Open-source.
  • EmbeddingsOpenAI text-embedding-3-small at $0.02/M tokens.
  • TaggingClaude Sonnet 4.6 via Anthropic API as the entity-extractor and relationship-tagger.
  • UpgradeWhen to migrate to enterprise tooling: only once the KB exceeds ~10 million validated data points or query latency exceeds 2 seconds. Not expected in Year 1.
04 / 06
Analyse
Auto · 24/7

Quantitative analysis engine · dual-AI consensus.

Claude API and ChatGPT GPT-5.5 API each independently run eight quantitative models per portfolio using data retrieved from the knowledge base. The outputs of the two engines are then reconciled. When both agree → VALID, forwarded to Step 5 as “Final Result”. When they disagree → PENDING, forwarded to Step 5 as “Requested Discussion”.

⚠ Models in this list are not yet finalized — subject to further discussion and revision
Recommended quantitative models · eight per portfolio
Long-Term Portfolio — Models 1 to 8
  • 1. Black-Litterman Model. Bayesian blending of market equilibrium with AI-generated views.
  • 2. Fama-French 5-Factor Model. Decomposes expected returns into market, size, value, profitability, and investment factors.
  • 3. Mean-Variance Optimisation (with Ledoit-Wolf shrinkage). Risk-adjusted weight solver.
  • 4. Discounted Cash Flow (with Reverse DCF cross-check). Intrinsic-value anchor for fundamental candidates.
  • 5. Monte Carlo Simulation (10,000+ paths, fat-tailed). Multi-year return distribution and drawdown analysis. Shared with short-term.
  • 6. Hidden Markov Model · regime detection. Identifies macro regimes to tilt allocation.
  • 7. Hierarchical Risk Parity (López de Prado). Stable allocation method that does not require expected-return inputs.
  • 8. LSTM / Transformer Forecast Ensemble. Deep-learning price and earnings trajectory forecasts.
Short-Term Portfolio — Models 1 to 8
  • 1. Monte Carlo Simulation. Shared with long-term; used here for Value-at-Risk and path-dependent scenarios.
  • 2. GARCH (1,1) / EGARCH. Volatility forecasting; primary driver of position sizing.
  • 3. Kalman-Filter Pairs Trading. Mean-reversion signal for cointegrated instruments.
  • 4. Ornstein-Uhlenbeck Mean-Reversion. Entry and exit timing for range-bound assets.
  • 5. Cross-Sectional Momentum (Jegadeesh-Titman 12-1). Ranks the candidate universe by trailing momentum.
  • 6. Time-Series Momentum (Moskowitz-Ooi-Pedersen). Trend-following overlay, especially effective on commodities and leveraged ETFs.
  • 7. XGBoost / LightGBM Classifier. Supervised signal combiner across technical and sentiment features.
  • 8. Fractional Kelly Criterion (capped). Optimal position sizing given edge and volatility; caps prevent catastrophic drawdowns.

Note: this list of models is not yet finalized — the team may add, remove, or replace models as the build progresses.

05 / 06
Discuss
Owner-in-the-loop

Discussion with owner · final decision gate.

When Step 4 reports a “Final Result” (both AI engines agree), the system provides a concise briefing and requests owner permission to execute. When Step 4 reports a “Requested Discussion” (the engines disagree), the system presents the full conflict — each engine’s reasoning, the specific point of divergence, and the data driving each view — and asks the owner for the final executive decision.

Channel · LINE / WhatsApp / Gmail

The owner receives briefings through a secure web dashboard plus an encrypted notification on their preferred channel: LINE (primary, given Thai operating context), with WhatsApp and Gmail as fallback channels. Each briefing includes model consensus status, proposed orders, confidence scores, worst-case Value-at-Risk, and one-click approve / reject / modify controls. For a 2–4 person team, this is a lightweight Notion or Linear-style interface rather than an enterprise trading desk.

06 / 06
Execute & Learn
Owner-in-the-loop

Execution & feedback loop.

Upon owner approval, orders are transmitted to the execution layer (broker selection pending) as normal cash-settled transactions — covering long-buying, short-buying, and leverage instruments (e.g. leveraged or inverse ETFs). No margin financing on the broker side and no direct derivative contracts (options, futures, swaps). The full result, successful or otherwise, is written back to the knowledge database with complete context: original plan, rationale, AI consensus level, execution quality, P&L, and root-cause tags on failures.

The loop never stops learning

A failed trade is not a loss — it is a labelled data point. Post-mortem analysis tags the failure type (timing, sizing, thesis, execution, external shock) and feeds it back into the Step 3 knowledge graph, so future runs of Step 4 weight similar setups accordingly.
