05 — AI Model Architecture & Voting System

Voting Architecture

Every model has an equal right to vote. All 15 models run independently and cast directional signals (YES / NO / NEUTRAL). A recommendation is triggered when ≥75% of models agree — meaning ≥6/7 for long-term and ≥6/8 for short-term. No model is supplementary; every result counts equally toward the final consensus.

Signal Confidence Tiers

Long-Term 7/7 or Short-Term 8/8100% — STRONG SIGNAL
Long-Term 6/7 or Short-Term 6–7/875–87% — RECOMMEND + REVIEW
Below 75% consensus<75% — NO SIGNAL

06 — Entry, Exit & Profit-Taking Rules
Long-Term Portfolio Rules ฿350K · 5 Positions Max

🎯 Entry Criteria

Position opened when ≥6 of 7 models vote YES (≥86% consensus). High-confidence 7/7 triggers auto-recommendation; 6/7 requires owner review before execution.

  • Allocation engines (L1, L2, L4): Must agree the asset fits within portfolio risk budget
  • Screening (L3, L7): Must show favorable factor exposure and fundamental value
  • Macro & risk (L5, L6): Must not flag extreme drawdown risk or unfavorable regime

🛑 Stop-Loss Rule (No Hard Stop)

No fixed percentage stop-loss. This is the authentic long-term investment philosophy: if the thesis and fundamentals remain valid, short-term volatility is noise.

  • Trigger at –20%: Mandatory review session — team + AI models + DBS consultation
  • Hold if thesis intact: If fundamentals and macro case still valid, hold position through drawdown
  • Exit only if thesis is broken: Fundamental deterioration, sector collapse, or macro regime shift invalidating original investment case
  • Discipline requirement: Reviews must be rigorous — not rationalizations for holding losing positions

💰 Profit-Taking Rule (Model-Driven)

No arbitrary profit targets. Positions held indefinitely as long as fundamentals support them. Exit only when the quantitative models signal overvaluation.

  • Exit trigger: L3 Fama-French flags negative factor exposure AND L7 DCF/DDM shows asset overvalued by >20% of intrinsic value
  • Quarterly rebalancing: Reallocates new capital and replaces exited positions only. Does not force profit-taking on winning positions.
  • Philosophy: Buy on consensus, hold on thesis, exit on valuation
Short-Term Portfolio Rules ฿150K · 5 Positions Max · ATR-Driven

🎯 Entry Criteria

Position opened when ≥6 of 8 models vote YES (≥75% consensus). Auto-execution for 6/8+ signals; below threshold queues for manual approval.

  • Trend/reversion (S1, S2): Must confirm directional edge
  • Technical + volatility (S3, S5): Must align for entry price and ATR-based stop distance
  • Catalyst/sentiment (S4, S8): Must not contradict trade direction
  • Risk sizing (S6): Position sized via Kelly Criterion adjusted by simulation confidence

🛑 Stop-Loss Rule (ATR-Based Dynamic)

Stop-loss set dynamically based on each asset's actual volatility, not a fixed percentage. Avoids the common flaw of stops too tight for volatile assets or too loose for stable ones.

  • Formula: Stop Price = Entry Price − (N × ATR)
  • N multiplier: Typically 1.5–2.5, calibrated by S5 GARCH volatility forecast
  • High GARCH vol forecast: Tighter N multiplier (protect from vol spikes)
  • Low GARCH vol forecast: Wider N multiplier (let normal fluctuations play out)
  • ATR source: S3 Technical Composite model already calculates ATR continuously

💰 Profit-Taking Rule (ATR Trailing Stop)

Let winners run. ATR-based trailing stop follows the price up, locking in gains as trend extends. Exits triggered when trend breaks.

  • Trailing stop: As price rises, trailing stop advances at the same ATR distance
  • Exit trigger 1: Price hits the trailing stop level (trend break)
  • Exit trigger 2: Model consensus flips — 6+ models shift to NO or REDUCE
  • No fixed profit target: No artificial cap at +15% or +20% — disciplined models + market volatility determine the exit
🔗 Model Synergy for Short-Term The short-term exit system is fully integrated with the voting models. S3 Technical Composite calculates ATR, S5 GARCH forecasts volatility and calibrates the N multiplier, and S6 Monte Carlo validates the risk/reward setup. This creates a self-consistent execution framework where entry, stop-loss, and exit all flow from the same model architecture.

  • Backup: All events also logged to Notion 2nd brain for historical reference
08 — Owner ↔ AI Discussion Framework

🗓 Discussion Triggers (Two-Mode)

Discussions happen in two modes: scheduled for routine review, triggered for high-impact decisions.

  • Scheduled weekly (every Friday): Aligned with team performance review. Covers portfolio health, trend analysis, model accuracy trends, strategic issues for the coming week.
  • Triggered ad-hoc: Long-term portfolio trade entries or exits, short-term trades exceeding capital threshold (to be set during Month 2–3 setup), signals where consensus is borderline, or supplementary context flagging caution.
  • Auto-execute without discussion: Routine short-term trades with ≥6/8 consensus below the capital threshold run on the standard auto-execution path.

📝 Tiered Documentation System

Documentation depth scales with decision significance. All tiers stored in Notion 2nd brain and tagged for feedback-loop analysis.

  • Tier 1 — Standard (routine): Decision + 2–3 sentence rationale. Used for most short-term trades, routine rebalancing, standard signal executions.
  • Tier 2 — Comprehensive (moderate): Decision + rationale + dissenting model views + explicit risk acknowledgment. Used for long-term trade entries/exits, above-threshold short-term trades, borderline consensus signals (6/7 or 6/8).
  • Tier 3 — Structured Template (high-stakes): Full standardized document with investment thesis, conviction level, expected catalysts, pre-defined exit conditions, dissenting views. Used for major capital allocation decisions, responses to DBS macro shifts, any decision affecting >20% of portfolio capital.

🤝 Disagreement Resolution — Re-Run With Context

Disagreements are never resolved by simple veto. Instead, the owner articulates what information or perspective the AI is missing, and the AI re-runs analysis with that new context.

  • Step 1: Owner articulates missing context (recent news, industry insight, personal risk factor)
  • Step 2: AI re-runs voting models with the new context fed in
  • Step 3a: If re-run confirms the owner's view → trade is revised accordingly
  • Step 3b: If re-run still supports original recommendation → escalate to DBS consultation for high-stakes cases, or log as a flagged disagreement in the feedback database for future review
  • Benefit: Creates a continuous learning system — every disagreement either improves AI inputs or becomes a documented case study
09 — Performance Tracking & Dashboard

🖥 Dashboard Stack — Hybrid Architecture

Three-tool hybrid, each serving a distinct purpose. Final architecture confirmed during Month 2–3 build phase before paper trading begins.

  • Notion — Team Hub: High-level portfolio summaries, weekly performance reports, meeting notes, decision logs. Natural fit with existing 2nd brain database.
  • Grafana — Real-Time Monitoring: Live time-series charts fed from TimescaleDB showing portfolio value, drawdown, model agreement trends. Best for technical monitoring.
  • Streamlit — Custom Analysis: Interactive analysis tool for ad-hoc queries (e.g., "show trades where GARCH flagged high vol," "compare model accuracy by month"). Built by team, flexible.

📊 Primary Performance Metrics

Five core metrics tell the complete performance story — measuring both outcomes and system quality.

  • Total Return vs Benchmark: Portfolio return vs 17.5% (long-term) and 25% (short-term) targets, and vs S&P 500. The bottom-line answer to "is this working?"
  • Sharpe Ratio: Risk-adjusted return. Distinguishes genuine skill from lucky volatility-induced gains.
  • Win Rate: Percentage of trades that are profitable. Execution quality indicator.
  • Model Accuracy: Percentage of voting signals that proved correct in hindsight. Validates whether the voting system actually predicts correctly. Critical metric — also drives the stop/pivot trigger in Section 15.
  • Trade Log: Full history with entry/exit rationale, model votes, discussion tier, and outcome. Audit trail + learning source.

📈 Secondary Metrics (Displayed but Smaller)

  • Maximum Drawdown: Peak-to-trough decline. Critical for triggering emergency DBS consultation if a serious drawdown occurs — always displayed even though not a primary focus.
  • Position-level P&L: Gain/loss per current holding. Informs quarterly rebalancing decisions.
  • Sector Exposure: Concentration risk across sectors and asset classes.

🗓 Review Cadence — Weekly Friday + Monthly Deep-Dive

Weekly rhythm aligned with short-term portfolio's weekly rebalancing cycle. Friday chosen for clean end-of-week review.

  • Every Friday (after market close): Full dashboard walkthrough, short-term trade log review, model accuracy check, action items for coming week
  • Last Friday of each month: Monthly deep-dive extends the Friday review — full trend analysis, long-term portfolio health, sector rotation assessment
  • Dashboard itself: Updates in real-time between reviews for anyone checking mid-week

11 — Contingency & Fallback Plans

🔄 AI Validation Disagreement (Claude vs Gemini)

When the two validation models cannot agree on data credibility, routing depends on data significance.

  • Significant data flagged for manual review: Fed announcements, earnings reports, major geopolitical events, news on currently held assets
  • Significance scoring layer: Auto-classifies each disagreement. High significance → human review queue. Low significance → auto-discard.
  • Reviewer action: Team member makes the final credibility call on flagged items. Decision logged for future AI training context.
  • Routine data on disagreement: Discarded automatically. No human review burden for minor items.

🔌 Data API Outage — Automatic Fallback Chains

Every critical data feed has a pre-configured fallback. If a primary API fails or returns stale data (no update for >5 minutes during market hours), the system automatically switches and notifies the owner.

  • Polygon.io → yfinance (market data)
  • Alpha Vantage → yfinance (technical indicators, reduced set)
  • GNews → RSS feeds + Grok API (news)
  • Nasdaq Data Link → Yahoo commodity feeds (commodities, reduced granularity)
  • Grok API → GNews + GDELT (real-time news)
  • If a fallback is also unavailable: System halts trades on affected assets until resolved. Full audit log entry created for every fallback event.

📉 Paper Trading Underperformance at Month 8

Tiered deployment based on actual demonstrated performance. All paths require DBS Phase 3 Capital Gate consultation before final commitment.

  • At or near benchmark (≥15% long-term / ≥22% short-term annualized): Deploy full capital as planned at Month 9
  • Moderate underperformance (10–15% long-term / 15–22% short-term): Deploy reduced capital (~50–70%) to test in live market while preserving runway. Revisit full deployment at next quarterly review.
  • Loss or flat returns (below 10% long-term / below 15% short-term, or negative): Delay live deployment. Extend paper trading 2–3 months, root-cause analysis with DBS, tune models, reassess.

🌐 DBS Macro Thesis Shift Mid-Project

DBS view is a critical input, but not an auto-execute trigger. Thesis changes prompt mandatory strategic review with owner holding final judgment.

  • Step 1: DBS communicates shift via private text chat during regular consultation
  • Step 2: Owner-AI strategic review session convened within 48 hours. L6 HMM Regime Detection and all macro-aware models re-run with the new thesis as input.
  • Step 3: Owner makes final call informed by three inputs — DBS view, AI model re-analysis, personal judgment
  • Step 4: Decision documented at Tier 3 (Structured Template) level given high-stakes nature
  • Step 5: Any portfolio pivot executed gradually, not all-at-once, to avoid whipsaw

16 — Project Success Criteria & Year 2 Pathway

🏆 Success Definition at Month 12 — Dual Condition

Project is considered successful at Month 12 only if both conditions are met. This multi-dimensional definition protects against misleading outcomes (either penalizing a tough market year where beating S&P 500 is a win, or rewarding profitability without system validation).

  • Performance condition: Combined portfolio return beats S&P 500 for the 12-month live trading period (or Month 9–12 live period pro-rated)
  • System condition: Model accuracy is demonstrably validated (voting signals proven correct at statistically meaningful rate), portfolio is profitable, execution has been clean (no major operational failures, no unplanned stop-loss breaches, no compliance issues)
  • One condition met only: "Conditional Success" — project continues but with mandatory adjustments based on Phase 5 Year-End review
  • Both conditions failed: Strategic reassessment with DBS to determine pivot or stop

🛑 Stop / Pivot Trigger — Persistent Model Failure

The primary stop/pivot trigger is model accuracy below 50% for 2 or more consecutive months. This focuses on system integrity rather than market outcomes — if models are consistently wrong, no amount of market luck will save the project.

  • Immediate action on trigger: Halt new trade entries for both portfolios. Existing positions continue under their exit rules.
  • Strategic review convened within 7 days: Full team + DBS consultation
  • Root cause analysis: Which models are failing and why? Bad data? Broken assumptions? Missed regime shift?
  • Decision paths: (a) Fix failing models and resume · (b) Pivot to a different strategy · (c) Pause project and preserve remaining capital
  • Capital loss and benchmark underperformance: Tracked but do not auto-trigger stops — they feed the strategic review as context

🚀 Year 2 Planning — Multi-Path Decision

Year 2 direction is decided at Month 12 Year-End Review (DBS Phase 5) based on the quality of Year 1 success. Year 2 scope is a Tier 3 documented decision (Structured Template) with full strategic review attended by all project members and DBS consultation.

  • Clean success path — Scale up: Year 1 is a clear success (beat S&P 500 comfortably + strong model accuracy + clean execution) → double capital for Year 2 to compound the demonstrated edge
  • Conditional success path — Optimize then scale: Year 1 succeeded but some models underperformed or specific weaknesses emerged → dedicate first 3–6 months of Year 2 to fixing identified weak models before adding any new capital
  • Open decision path — Decide at Month 12: Results are ambiguous or market conditions shifted significantly → keep all options open until Phase 5 Year-End consultation. Combined perspective of Year 1 data + current macro view + updated 3-year DBS outlook informs final Year 2 direction
17 — Key Recommendations & Implementation Notes

🏗 Architecture & Tech Stack

  • Use Python as the primary language — best ecosystem for quant finance (pandas, numpy, scipy, yfinance, QuantLib, statsmodels, arch, hmmlearn, transformers)
  • Use Apache Airflow or n8n for orchestrating automated pipeline runs (scheduling, retry logic, alerting)
  • Store raw data in PostgreSQL with TimescaleDB extension — time-series optimized and scales well with market data volume
  • Use Redis for caching real-time price feeds to avoid redundant API calls
  • Build dashboard with Notion + Grafana + Streamlit hybrid (finalized Month 2–3)

🤖 AI Strategy

  • Use Claude Opus for complex relationship mapping and deep analysis; use Claude Haiku for routine data categorization (cost efficiency)
  • Use xAI Grok API as the primary real-time news feed — its live X/Twitter data access makes it uniquely fast for detecting breaking market-moving events for short-term signals
  • Use structured output (JSON mode) from Claude for quant model inputs — avoids parsing errors
  • Log every AI decision with input context + output reasoning for audit trail and future model improvement — this feeds directly into the Notion 2nd brain

⚠ Critical Cautions

  • Paper trade Month 3–8 minimum before deploying real capital — validate model accuracy against live market conditions
  • The 25%/yr short-term target is aggressive — most hedge funds target 15–25% and sit near the top. Combined with 5-position concentration, single bad picks can hurt. DBS Phase 3 consultation is the final sanity check.
  • Regulatory & broker access: Short-term broker selection (Month 1–2) must verify Thai resident compatibility and API access. Long-term manual execution needs no special broker setup.
  • Keep AI analysis as a decision support tool, not full automation for long-term decisions. Short-term auto-execution is acceptable because the ≥6/8 consensus threshold pre-filters low-confidence trades.
  • Liability boundary: All trading decisions and outcomes are owner responsibility. DBS provides advisory input only.

💡 Quick Wins (Start Here in Month 1)

  • Start with yfinance + FRED + SEC EDGAR + GDELT — all free, easy to integrate, cover the core data needs while premium APIs are being set up
  • Use Notion as the 2nd brain from day one — low overhead, team-friendly, solid API. Migrate to Neo4j only if relationship complexity genuinely grows in Year 2.
  • Build the long-term portfolio system first (simpler, less time-sensitive) before tackling the short-term high-frequency system
  • Schedule DBS Phase 1 kickoff consultation in Month 1 before too many architecture decisions are locked in — their macro view may shape priorities

📝 Documentation Standards

  • Every AI-generated recommendation must log: input context, all 15 model votes (or 7/8 per portfolio), consensus level, confidence tier, and any supplementary caution flags
  • Every owner-AI discussion must produce Tier 1 / 2 / 3 documentation per Section 08, stored in Notion with consistent tagging
  • Every trade execution must log: entry price, target if applicable, stop level (ATR-based for short-term), model rationale, actual exit price, realized P&L
  • Failures are as valuable as successes in the feedback database — document "what went wrong" with full honesty, not rationalizations
